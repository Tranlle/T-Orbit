# Subtitle Translator 插件任务拆单

## 1. 文档目的

本文档基于 [SubtitleTranslator.Plugin.Plan.md](C:\Users\tangj\Desktop\project\T-Orbit\SubtitleTranslator.Plugin.Plan.md) 的开发规划，将 `Subtitle Translator` 插件拆分为可执行的开发任务。

目标不是做泛泛的阶段说明，而是把后续开发工作压缩成：

- 可以逐项领取的 Todo
- 明确输入输出的交付项
- 可判断完成与否的验收条件
- 可用于安排开发顺序的依赖关系

## 2. 插件范围回顾

首版目标：

- 手动框选字幕区域
- 周期性截取选区画面
- 本地 OCR 识别
- OCR 图像预处理
- 字幕去重与稳定化
- 原文实时显示
- 译文实时显示
- 开始、暂停、重新选区
- 运行日志与错误提示

本拆单文档只覆盖首版和首版前的原型验证，不覆盖增强阶段的悬浮窗、自动识别字幕区、快捷键等功能。

## 3. 总体执行顺序

推荐执行顺序：

1. Milestone 1：原型验证
2. Milestone 2：MVP 闭环
3. Milestone 3：可用性优化
4. Milestone 4：增强能力预留

执行原则：

- 先验证 OCR 可行性，再做界面细化
- 先做业务闭环，再补调参体验
- 先解决流水线稳定性，再扩展功能范围

## 4. 任务状态建议

后续跟踪任务时，建议每项任务使用以下状态：

- `Todo`
- `In Progress`
- `Blocked`
- `Done`
- `Deferred`

这是当前最稳的顺序，因为它把最大不确定性放在最前面：

- OCR 能不能稳定跑
- 截图链路能不能闭环
- 稳定化能不能压住抖动

这些问题如果不先解决，后面的 UI 精修和高级功能都没有意义。

## 10. Definition of Done

每项任务在标记为 `Done` 前，至少满足以下条件：

1. 代码已编译通过
2. 不破坏现有宿主与其他插件
3. 对应路径有基本自测
4. 异常情况下不会直接白屏或崩溃
5. 页面或日志能反映任务结果

## 11. 结论

这份拆单文档的核心目的，是把 `Subtitle Translator` 的开发工作从“有方向”推进到“可按项执行”。

后续真正开始开发时，可以直接以 `M1-01` 到 `M2-10` 为主路径推进。  
只要原型验证通过，这个插件的首版实现路径就是清晰且可控的。
